## 🧠 Thinking
- 今天学到什么（主要是写 CODEBASE_TOUR 过程中啃下来的）：
  - **Ingest 流程**：`docs/*.txt` → 正则解析头部（Title/Link/Instructor）→ 状态机扫 `Lesson N:` 分段 → 按句子切片（≤800 字符，整句重叠 ≤100，绝不切断句子，超长单句独占 chunk）→ 加前缀 `Course X Lesson N content:` → `all-MiniLM-L6-v2` 生成 384 维向量 → ChromaDB 双 collection（`course_catalog` 存元数据 + `course_content` 存 chunk），物理落到 `chroma.sqlite3`（原文 + metadata + FTS5）+ `<segment-uuid>/*.bin`（HNSW 图）
  - **Chunk 前缀的作用**：每个 chunk 单独带课程/课时锚点，单看一个 chunk 也有语义定位，提升检索质量
  - **Agent workflow = 多轮 tool-use 循环**：`messages` 累加 → Claude 返回 `stop_reason="tool_use"` → 执行工具把 `tool_result` 塞回 messages → 下一轮 Claude 基于上轮结果决定再搜还是出答案 → 预算耗尽做一次不带 tools 的兜底
  - **检索智能在模型不在后端**：没有任何 `if/else` 决定怎么搜，全靠 tool schema 的自然语言描述 + system prompt 指引 + Claude 的语义理解
  - **双 collection 设计是领域驱动**：课程天然有"课程级"和"chunk 级"两个粒度，所以有 catalog；扁平语料没必要
  - **没 eval 先别动手**：先搭评估再做改动，三次改动全靠它做回归 gate

## ⚡ Decisions
- Ground truth 全部 Claude 合成，不做人工标注（靠抽检缓解泄漏）
- MRR 而非 NDCG（每题只有一个正确课时，相关性二元）
- 多轮硬上限 N=2；While 循环完全替换旧单轮分支；sources 追加 + 去重
- 为什么：最小可用 eval 比完美 eval 更重要；MRR 足够本项目场景；N=2 是"够自救 + 不炸费用"的甜点
